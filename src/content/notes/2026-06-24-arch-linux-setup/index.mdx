In 2018 I wrote a [two](../2018-01-24-arch-linux-installation-guide-part-1/) [part](../2018-01-26-arch-linux-installation-guide-part-2/) guide on installing Arch Linux. It was a step by step command reference. Every step I took then was manual. I read the wiki, typed the command, and fixed whatever broke. It taught me a lot about Arch Linux and Linux in general.

I did it again this week, on the same old HP laptop. This time, I wanted to turn this laptop into a headless home server. This time however, I did not type most of the commands myself. An AI coding agent called [Pi](https://pi.dev/) managed the installation for me. The experience was faster than 2018, but it was not smooth. The agent made quite a few mistakes. Some were harmless. Some were dangerous. And a few things I had to do myself because the agent simply could not.

This post covers what happened – what the agent got right, what it got wrong, and where I still needed to be in control.

## The push to start

I booted the laptop from the Arch ISO, set a root password on the live environment, and used `ssh-copy-id` to copy my SSH key over. From there, the agent connected over SSH and handled the entire installation remotely.

## Partitioning

The laptop has a 500 GB SSD with Windows already on it. About 250 GB was free at the end of the disk. The agent analyzed the partition table and showed me the exact boundaries.

| Partition | Size  | Purpose              |
|-----------|-------|----------------------|
| sda1      | 100M  | EFI (existing)       |
| sda2      | 16M   | Microsoft reserved   |
| sda3      | 214G  | Windows C:           |
| sda4      | 650M  | Windows recovery     |
| sda5      | 250G  | New, for Arch        |

I confirmed the plan and the agent created the partition with `fdisk` and formatted it with btrfs.

```bash
fdisk /dev/sda
# created a new partition in the free space
mkfs.btrfs -f /dev/sda5
```

## btrfs subvolumes

Once the partition was formatted, things went smoothly. The agent created four subvolumes.

- `@` for root
- `@home` for home directories
- `@var_log` for logs
- `@snapshots` for snapshots

It mounted everything with zstd compression. It ran pacstrap to install the base system. The first pacstrap timed out because of download speed. The agent retried it, but the partial install left a lock file. It cleaned that up and retried successfully.

## Bootloader confusion

The agent started with systemd-boot. That was a mistake because the EFI partition was only 100 MB. When systemd-boot copied kernel files to EFI, the partition filled up. The agent switched to GRUB, which only needs a small binary on EFI and reads kernels from the btrfs root.

The switch itself was clean. But the earlier systemd-boot attempt left kernel files on EFI that had to be manually removed. The agent figured this out and cleaned them up.

After installing GRUB with os-prober, the grub config only showed Windows. The kernel files were missing from /boot. The earlier pacstrap timeout had interrupted the kernel package download. The agent reinstalled the kernel and regenerated the config. Both Arch and Windows appeared in the boot menu.

## WiFi fix

The laptop has a Realtek RTL8723BE WiFi card. These cards are notorious on HP laptops because the antenna is wired to the wrong connector. The fix is a kernel module parameter.

The agent set `ant_sel=2` first. The signal improved but was still weak. It scanned the networks again, saw that signal was still poor, and tried `ant_sel=1`. That gave full signal strength and over ten visible networks. It wrote the setting to a config file so it persists across boots. The fix is identical to what I manually applied via my GitHub repo years ago.

Fixing the WiFi was one of the agent's strongest moments. It tried an approach, measured the result, and adjusted based on what it saw.

## Static IPs and interface names

I wanted static IPs for both ethernet and WiFi. The agent set them up with NetworkManager. Later, after a reboot, the interface names changed. The ethernet went from `eno1` to `eth0` and the WiFi from `wlo1` to `wlan0`. The NetworkManager profiles were bound to the old names, so they stopped working. The static IPs were gone and the WiFi would not connect.

The agent found the root cause. A systemd link file that it had created for Wake on LAN was overriding the default naming policy. It fixed the link file, updated the NetworkManager profiles, and rebuilt the initramfs. After another reboot, the names were stable.

The WiFi password was also lost during this. The agent could not enter it because NetworkManager needs a TTY for secrets. I had to type the password myself on the laptop console.

## Server tuning

I decided to use this laptop as a headless server, not a desktop. No screen, no keyboard. Just SSH. The agent did a lot of configuration here.

It disabled the lid switch so closing the laptop does nothing. It masked sleep, suspend, and hibernate targets. It set the CPU governor to performance and lowered swappiness from 60 to 10. It installed TLP for power management. It capped journal logs at 500 MB. It enabled weekly fstrim for the SSD. It set up pacman cache cleanup and reflector for fast mirrors.

When I closed the lid to test, the agent happily reported it was still connected.

## Battery alerts

I wanted Discord notifications when the battery drops below 25 percent or goes above 80 percent. The agent wrote a script that reads the battery level from sysfs every two minutes. It used apprise to send the notifications.

The script and a systemd timer were set up without issues. The agent also set up a weekly update script that sends a Discord notification with the package list, runs the update, and reports the result.

## Shell and development tools

For the terminal, the agent tried several console fonts. I did not like any of them. I eventually settled on the same font that Ubuntu Server uses. The agent also had problems with the autologin configuration. It used the wrong path for agetty, so the login prompt disappeared and kernel debug messages filled the screen. It fixed this after checking the correct binary path.

The shell setup included autosuggestions, syntax highlighting, and history substring search. The agent added battery percentage to the prompt. The emojis it tried to use did not render on the console font, so it switched to text indicators.

For development tools, the agent installed Docker, Go, Python, and Node.js. It used version managers where possible so everything is sudoless. The AUR helper yay was installed after paru failed because of a library version mismatch. The agent tried to install the NVIDIA drivers too, but the laptop's 940M GPU is too old for the open source driver and the proprietary package no longer exists in the Arch repositories. After a few failed attempts, it cleaned up and moved on.

## Wake on LAN

Wake on LAN works over ethernet, but only when the charger is plugged in. The ethernet port loses power on battery after shutdown. This is a hardware limitation on most consumer laptops. The agent set up the ethtool commands and a udev rule to keep it enabled. From my macOS machine, I can wake the laptop with a single command, as long as the charger is connected.

## What the agent could not do

The agent could not enter passwords that required a TTY. I had to type the WiFi password on the console. It could not change BIOS settings for Wake on LAN. It could not physically plug in the charger or press the power button.

It also could not know when to stop. The "device busy" loop was the clearest example. The agent tried over a dozen approaches to format a partition that the kernel refused to release. Some of its ideas were creative. Most were variations of the same thing. A human would have rebooted after the third attempt.

The agent also tried to delete all partitions on the disk at one point. It backed off when I told it to. But the fact that it even considered it, given that the disk held Windows, was concerning. I had to stay alert.

## What changed since 2016

In 2018, every step was manual. I read the wiki, typed the command, fixed the error, and repeated. It took a weekend. But I understood every decision. Nothing happened that I did not intend.

With the agent, the speed was higher, but my attention had to shift from doing to supervising. The agent executed most commands correctly. When it made mistakes, they were sometimes subtle (wrong config path) and sometimes alarming (partition wipe). I had to catch them.

The installation took a couple of hours overall. In 2018, it took a weekend. The time saving came from the agent doing the mechanical work: creating partitions, running pacstrap, configuring systemd units, writing scripts. I focused on the decisions: btrfs or ext4, GRUB or systemd-boot, which packages to install.

Would I use an agent again for this? Yes. But I would not leave it unattended. It is a fast assistant that needs a watchful human.
